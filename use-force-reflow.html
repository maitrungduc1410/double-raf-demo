forceReflow — kích hoạt transition mà KHÔNG cần rAF

Đây là cách react-transition-group làm. Cả hai ô dưới đây đều đổi từ trạng thái A sang B đồng bộ trong cùng một task — không dùng requestAnimationFrame, không setTimeout. Khác biệt duy nhất: ô bên phải đọc scrollTop ở giữa, ép trình duyệt vẽ lại ngay lập tức.

Trình duyệt—
Engine—

❌ Không forceReflow

Set class .from rồi set .to ngay lập tức. Trình duyệt gộp 2 trạng thái thành 1 paint → mất transition.

—

✅ Có forceReflow

Set .from, đọc node.scrollTop (ép reflow đồng bộ), rồi mới set .to. Trạng thái A được chốt trước → transition chạy.

—
// Box vừa được tạo ở trạng thái .from (opacity 0) // Cả hai đều chạy đồng bộ, KHÔNG có rAF / setTimeout // ❌ Không forceReflow — bị gộp paint box.classList.remove('from'); box.classList.add('to'); // đổi luôn → trình duyệt gộp lại // ✅ Có forceReflow — transition chạy void box.scrollTop; // đọc layout = ép reflow, chốt .from TRƯỚC box.classList.remove('from'); box.classList.add('to'); // giờ .from → .to là 2 paint khác nhau

Vì sao chỉ đọc một thuộc tính lại "cứu" được transition? Khi bạn đọc scrollTop (hay offsetHeight, getComputedStyle...), trình duyệt buộc phải tính toán style + layout ngay tại đó để trả về giá trị đúng — gọi là forced synchronous layout. Nhờ vậy trạng thái .from được "chốt" lại như một mốc riêng, nên khi bạn đổi sang .to, trình duyệt thấy đây là hai trạng thái khác nhau và chạy transition. Khác với 1× rAF, cách này hoạt động đúng trên mọi engine — kể cả Firefox — vì nó không phụ thuộc vào thời điểm rAF fire.
